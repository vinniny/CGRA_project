ABSTRACT
This project presents the specification and architectural definition of a CoarseGrained Reconfigurable Architecture (CGRA) accelerator optimized for sparse Spiking
Neural Network (SNN) inference. Modern deep learning workloads—particularly
event-driven SNNs—are heavily constrained by the “memory wall” due to irregular
data access patterns and high weight sparsity. To address these bottlenecks, the
proposed accelerator employs a 4×4 mesh of Processing Elements (PEs) interconnected
through a multicast-capable routing fabric, enabling efficient execution of Compressed
Sparse Column (CSC) operations.
A distinguishing feature of this architecture is the integration of a dedicated
Leaky Integrate-and-Fire (LIF) neuron update unit within each PE, allowing
biologically inspired neural dynamics to be computed directly in hardware. The system
targets deployment on Xilinx Zynq UltraScale+ XCZU4EV, leveraging the ARM
Cortex-A53 Application Processing Unit (APU) for control and orchestration, while
delegating high-throughput computation to the FPGA fabric. Communication between
the subsystems is implemented through AMBA AXI4 for high-bandwidth DMA
transfers and APB for configuration and control.
Key innovations include a double-buffered configuration memory enabling
seamless, low-latency context switching, and a deterministic spatio-temporal
scheduling mechanism that enhances utilization and predictability. Together, these
features form a flexible and efficient hardware platform tailored for real-time, powerefficient SNN inference.

1. INTRODUCTION
1.1.

Overview of CGRA and SNN Inference
This specification presents the architecture and design methodology of a 4×4

fixed-point Coarse-Grained Reconfigurable Array (CGRA) accelerator, developed
as a domain-specific hardware engine for sparse Spiking Neural Network (SNN)
inference. Unlike conventional neural networks that rely on dense tensor operations,
SNNs operate on sparse, event-driven computations dominated by irregular memory
access patterns and lightweight neuronal dynamics. These characteristics introduce
fundamental inefficiencies in traditional von Neumann processors and GPU-based
accelerators, where SIMD utilization drops sharply with sparsity.
The proposed CGRA addresses these limitations through a spatially
programmable 4×4 mesh of Processing Elements (PEs), each combining a fixedpoint Multiply–Accumulate (MAC) unit with a hardware-implemented LeakyIntegrate-and-Fire (LIF) neuron update engine. This unified MAC+LIF execution
model enables the array to compute sparse matrix–vector (matvec) operations and
neuron dynamics in a tightly coupled, dataflow-driven manner. Execution is
governed by cycle-accurate configuration words stored in per-PE context memories,
enabling temporal scheduling similar to VLIW-style instruction flows while
maintaining deterministic timing.
The accelerator is deployed within the Programmable Logic (PL) of the
Xilinx Zynq UltraScale+ XCZU4EV MPSoC, leveraging the tight integration
between the hardware logic and the ARM Cortex-A53 Processing System (PS). The
PS serves as the system controller and orchestrator, managing configuration, DMA
programming, input data movement, and post-processing. Communication between
the processor and the accelerator is facilitated via the General Purpose AXI Master
interface (M_AXI_HPM) using memory-mapped AXI4-Lite Control and Status
Registers (CSRs). Bulk data transfer—such as reading CSC-formatted columns or
writing updated neuron states—is performed via an AXI4 High-Performance (HP)
slave interface connected to shared DDR4 memory. [1]

2. LITERATURE REVIEW AND BACKGROUND
2.1.

Overview of Coarse-Grained Reconfigurable Architectures (CGRAs)
Coarse-Grained Reconfigurable Architectures (CGRAs) represent a class of

programmable hardware structures positioned between traditional fine-grained
FPGAs and fixed-function ASICs. Unlike LUT-based FPGA fabrics, CGRAs are
composed of word-level functional units, called Processing Elements (PEs),
organized in a regular array. Each PE typically includes arithmetic logic capabilities,
local register files, and configurable routing paths that facilitate parallel
computation.
Modern machine learning and signal processing algorithms frequently
demand high throughput and reconfigurable computation patterns. Fine-grained
FPGAs can satisfy these requirements but suffer from high configuration overhead,
routing congestion, and inefficient mapping of arithmetic-heavy workloads. CGRAs
mitigate these issues by operating at the arithmetic-operator granularity, eliminating
much of the redundant configuration and interconnect complexity found in LUTbased architectures. This provides improved energy efficiency, deterministic
performance, and reduced implementation effort for many computational kernels.
A typical CGRA executes computations in three primary phases:
1. Configuration Mapping – Where context words define the operation of
each PE.
2. Spatial/Temporal Scheduling – Where operations are mapped to
specific cycles and PEs.
3. Data Routing and Execution – Where inputs flow through the mesh
according to the configured routing scheme.
This organization allows the CGRA to support complex, multi-stage
computations without the excessive reconfiguration that limits traditional FPGAs.
The fundamental advantage lies in its ability to execute nested loops, dataflow

2.2.1. Overview of the CGRA Processing Element

proposed design, each PE integrates:

Spiking Neural Networks (SNNs)
Spiking Neural Networks (SNNs) represent the third generation of neural

network models, in which information is encoded and processed using discrete spike
events rather than continuous-valued activations. Unlike traditional Artificial Neural
Networks (ANNs) that operate on static real-valued vectors, SNNs introduce an
explicit notion of time, and neuronal state evolves across discrete timesteps
according to dynamical equations.
This temporal and event-driven formulation allows SNNs to more closely
resemble biological neural systems, where neurons communicate through short
electrical impulses (spikes), and computation is often sparse in both time and space.
From a hardware perspective, these characteristics are attractive because they enable
data-dependent, sparse computation, which can reduce switching activity and
energy consumption if mapped onto an appropriately designed architecture. [5] [6]

2.3.1. Neuron State, Membrane Potential, and Spike Events
In SNNs, each neuron maintains a membrane potential v(t), which is a state
variable summarizing the neuron’s internal condition at time t. The membrane
potential integrates incoming synaptic currents and decays (or “leaks”) over time.
When v(t) exceeds a firing threshold Vth, the neuron emits a spike and its membrane
potential is typically reset. [4]
At a high level, the behavior of a neuron can be separated into three phases
per timestep:
1. Integration:
Incoming spikes from presynaptic neurons cause weighted increments to
the membrane potential.
2. Leak:
The membrane potential decays over time to model passive leakage of
charge.
3. Threshold and Reset:
If the membrane potential exceeds a threshold, a spike is generated and
the potential is reset (often to a baseline value).
In discrete-time SNN models used for digital hardware, these phenomena are
approximated with simple difference equations, making them suitable for fixedpoint arithmetic and parallel implementation. [6]
2.3.2. Leaky Integrate-and-Fire (LIF) Neuron Model
Among many neuron models proposed in computational neuroscience, the
Leaky Integrate-and-Fire (LIF) model strikes a balance between biological
plausibility and implementation simplicity. It captures the essential features of
integration, leakage, and threshold-based spiking without the complexity of
biophysical models such as Hodgkin–Huxley.

2.3.3. Information Encoding Schemes
Unlike Artificial Neural Networks (ANNs) where information is transmitted
as continuous floating-point values (activations), SNNs communicate via binary
spike events. Therefore, a method is required to encode real-world data (such as
pixel intensities in an image) into a train of spikes. There are two primary encoding
paradigms:
 Rate Coding (Frequency Coding):
Information is conveyed by the firing rate of the neurons. A higher input
value corresponds to a higher frequency of spikes over a given time window. While
robust and easy to implement, rate coding can be inefficient as it requires a longer
simulation time to estimate the precise value of a signal. [5]
 Temporal Coding (Time-to-First-Spike):
Information is encoded in the precise timing of individual spikes relative to
a stimulus onset. For example, a stronger stimulus might cause a neuron to fire
earlier than a weaker stimulus. This method is highly sparse and energy-efficient, as
it relies on fewer spikes to transmit the same amount of information, making it
particularly attractive for hardware accelerators aiming for ultra-low latency. [6]

In the context of this CGRA accelerator, the architecture is designed to be
encoding-agnostic, supporting sparse spike vectors regardless of whether they were
generated via rate or temporal schemes, provided they are formatted into the
appropriate Compressed Sparse Column (CSC) representation.
2.3.4. Training Methodologies for SNNs
Since this project focuses on a hardware accelerator for inference, it is
essential to understand how the parameters (weights and thresholds) are obtained.
There are two main approaches to training the SNNs that this accelerator will
execute:
1. ANN-to-SNN Conversion:
A conventional ANN is trained using standard backpropagation on a GPU. Once
trained, the floating-point weights are mapped to SNN parameters. The
activation of an ANN neuron is approximated by the firing rate of an SNN
neuron. This method achieves high accuracy on complex tasks (like ImageNet)
but may require higher latency during inference to approximate the values
accurately. [8] [9]
2. Direct Training (Spike-Based Learning):
The SNN is trained directly in the spiking domain. This can be done using:
o

Surrogate Gradients: Approximating the non-differentiable spike function
to allow backpropagation through time (BPTT).

o

Unsupervised Learning (STDP): Spike-Timing-Dependent Plasticity is a
local learning rule where the weight of a synapse is adjusted based on the
relative timing of the presynaptic and postsynaptic spikes. If the presynaptic
neuron fires just before the postsynaptic neuron, the connection is
strengthened (Long-Term Potentiation).

The proposed CGRA accelerator is capable of executing SNN models trained
via either method, as the resulting weights are loaded into the system memory prior

2.3.5. Computational Advantages over Von Neumann Architectures
The theoretical motivation for designing a specific CGRA for SNNs lies in
the fundamental differences between neuromorphic and Von Neumann
computations:


Event-Driven Processing: In standard matrix multiplication (used in ANNs),
every weight is multiplied by every activation, even if the activation is zero (or
near zero). In contrast, SNNs are event-driven; computation in the LIF unit is
only triggered when an input spike arrives. If no spike occurs, the PE remains
idle, saving dynamic power.



Sparsity Exploitation: Biological neural activity is incredibly sparse (often
<1% activity). By using CSC format and a multicast routing network, this
architecture avoids fetching "zero" values, overcoming the "Memory Wall"
bottleneck that limits the performance of standard CPUs and GPUs on sparse
workloads. [11]



Locality of Reference: The LIF neuron model requires storing the membrane
potential v[t] locally. In a CPU/GPU, this state must be constantly shuffled back
and forth to main memory. In this CGRA, the state is kept in the local PE
scratchpad (SPM), drastically reducing data movement energy. [6]

2.3.6. Energy Efficiency and Event-Driven Processing
The fundamental advantage of SNNs lies in their event-driven nature, which
contrasts sharply with the dense, frame-based processing of traditional ANNs. In
ANNs, every neuron in a layer computes an activation value every cycle, regardless
of whether the information is relevant. SNNs, however, are quiescent by default;
they consume dynamic power only when a spike event occurs.

19

Capstone Project 2

Instructor: Assoc.Prof, PhD.Truong Quang Vinh

This characteristic aligns naturally with the "sparsity" observed in biological
systems and modern neuromorphic sensors (like DVS or SPADs), where data is
sparse in both time and space. By processing information only upon the arrival of
relevant input spikes, SNNs can achieve significant reductions in switching activity
and energy consumption. For hardware accelerators like the proposed CGRA, this
allows for fine-grained clock gating or logic disabling during idle periods between
spikes, overcoming the "memory wall" and power constraints typical of edge
computing environments. [5] [6]
2.3.7. Challenges in SNN Training and Deployment
Despite their efficiency, deploying SNNs presents unique challenges compared to
standard deep learning:


Training Complexity: Unlike ANNs which use standard backpropagation,
SNNs require specialized training methods due to the non-differentiable nature
of discrete spikes. Common strategies include ANN-to-SNN Conversion
(converting a pre-trained ANN to SNN, often at the cost of higher latency) and
Direct Training using surrogate gradients (approximating the spike function to
enable backpropagation through time). [5]



Hardware Mismatch: Standard hardware is rarely optimized for sparse,
irregular memory access patterns inherent in SNNs. This "algorithm-hardware
mismatch" necessitates custom architectures - like the CGRA proposed in this
project - that support efficient multicast routing and local state management to
fully exploit the theoretical efficiency gains of SNNs. [7]

The internal architecture of the CGRA is partitioned into four major subsystems,
designed to maximize data throughput and minimize stall cycles:
1. Global Control and CSR: This module acts as the interface to the host
processor. It manages the accelerator's lifecycle, handling interrupt generation,
status monitoring, and the decoding of commands via the APB slave interface.
2. Double-Buffered Configuration Memory: To support rapid reconfiguration
required by multi-layer neural networks, the configuration memory employs a
double-buffering scheme. This allows the DMA to pre-fetch the bitstream for the
next layer into the "back" buffer while the current layer is executing from the
"front" buffer, effectively hiding reconfiguration latency.
3. DMA Engine and Bank Selector: The Direct Memory Access (DMA) unit
operates as an AXI4 Master to fetch data from external DRAM. The Bank
Selector logic routes these incoming data streams to the appropriate on-chip
memory banks based on the target row index, ensuring efficient data scattering.

4. Row-Banked Tile Memory: The global tile memory is physically partitioned
into four independent SRAM banks (Bank 0 to Bank 3). Each bank is dedicated
to a specific row of the PE array (e.g., Bank 0 serves Row 0). This Row-Banked
organization eliminates memory contention, enabling all four rows of PEs to
fetch operands simultaneously in a single clock cycle.
5. 4×4 PE Array: The computational core consists of 16 Processing Elements
arranged in a 2D mesh topology. Each PE (indexed 0,0 to 3,3) is tightly coupled
with the routing logic to perform parallel neuromorphic computations.

5. VERIFICATION AND TESTING
Ensuring the functional correctness of the CGRA accelerator is critical, given
the complexity of the 4x4 mesh interconnect, the distributed LIF neuron updates,
and the asynchronous DMA transactions. To guarantee production-quality, a
robust, comprehensive, layered verification environment was developed using
SystemVerilog and C++ co-simulation.
5.1.

Verification Methodology

The verification strategy transitions from simple directed tests to a Constrained
Random Verification (CRV) approach. This methodology is essential for
uncovering corner-case bugs that manual testing often misses, such as pipeline
hazards, bus contention, and arithmetic overflows.
The verification environment relies on a Self-Checking Architecture composed
of three subsystems:
 Stimulus Generator (Driver): Produces randomized inputs (spike vectors,
weights, configuration frames) constrained to valid ranges.
 Golden Reference Model: A high-level behavioral model that runs in parallel
with the Design Under Test (DUT).
Scoreboard: A centralized checker that compares the DUT's output against the
Golden Model's prediction in real-time.
5.2.

Golden Model Architecture

The Golden Model serves as the source of truth for verification. Unlike the RTL
(Register Transfer Level) code, which describes how the hardware is built (gates,
flip-flops, wires), the Golden Model describes what the hardware should do
(mathematical behavior).
 Behavioral Abstraction: The model computes results using 64-bit precision
to detect overflows before casting to the 32-bit target width. For example, the
saturation logic for arithmetic operations (like ADD and MAC) is
implemented as follows:

The verification environment is structured as a layered testbench to maximize

The Test Layer At the top of the hierarchy is the Test Layer, responsible for
defining and orchestrating the verification scenarios.
 Test Case: This component initiates the simulation process. It holds the
specific objectives of a given test run and determines pass/fail criteria based
on feedback from lower layers.
 Test Sequencer: The Test Case passes high-level scenario requirements down
to the Test Sequencer. The Sequencer decomposes these requirements into
sequences of individual, abstract transactions to be executed.
The Driver/Monitor Layer The middle layer acts as a bridge between abstract
transactions and concrete signal activity. It is split into stimulus generation
(Driver) and observation (Monitor) paths.
 Driver (BFM): The Driver, often referred to as a Bus Functional Model

responsible for converting these transactions into the specific pin-level signal
protocols required to communicate with the hardware.
 Monitor (Checker/Scoreboard): Parallel to the driver, the Monitor passively
observes the signals on the interface. It reconstructs signal activity back into
abstract transactions. These observed transactions are typically passed to an
internal or external Checker/Scoreboard to verify protocol compliance and
data integrity against expected results.
The Interface/DUT Layer The bottom layer represents the physical connection
to the hardware being verified.
 System Interface: This block represents the actual bundling of signals, clocks,
and resets connecting the testbench to the design. The Driver actively drives
signals onto this interface, while the Monitor passively samples signals from
it.
 Design Under Test (DUT): This is the actual hardware logic being verified.
It receives stimulus via the System Interface, processes it, and drives
responses back onto the System Interface.
Execution Flow The overall data flow forms a closed loop:
1. The Test Case defines a scenario.
2. The Test Sequencer generates transaction sequences.
3. The Driver converts transactions into signal stimuli on the System Interface.
4. The DUT receives stimuli and drives responses back to the System Interface.
5. The Monitor observes these responses on the interface.
6. Results from the Monitor/Scoreboard are passed back up to the Test Case for
final evaluation.

7. CONCLUSION AND DEVELOPMENT DIRECTION
7.1.

Conclusion

This project successfully designed and implemented a Coarse-Grained
Reconfigurable Architecture (CGRA) accelerator tailored for sparse Spiking
Neural Network (SNN) inference. The architecture addresses the fundamental
inefficiencies of von Neumann processors in handling irregular, event-driven
workloads by integrating a 4x4 mesh of neuromorphic Processing Elements.
Key achievements include:
1. Hardware-Software Integration: A fully functional AXI4-based system that
allows the ARM Cortex-A9 HPS to offload compute-intensive kernels to the
FPGA fabric.
2. SNN-Specific Features: The integration of hardware LIF neuron updates and
multicast routing directly accelerates the biological dynamics of spiking
networks.
3. Seamless Reconfiguration: The double-buffered configuration memory
enables "hot-swapping" of layers, hiding reconfiguration latency and
maintaining high PE utilization.
4. Robust Verification: With 100% pass rate on 6169 test vectors (March 2026),
zero failures, and zero protocol violations, the IP is robust and ready for
physical deployment.
7.2.

Development Direction
Although the proposed CGRA accelerator has demonstrated correct

functionality and promising architectural advantages for sparse Spiking Neural
Network (SNN) inference, several extensions and improvements can be explored in
future work to further enhance its performance, flexibility, and applicability.
First, the current implementation focuses primarily on inference workloads
with statically configured weights and neuron parameters. A natural extension is to
support on-chip learning mechanisms, such as Spike-Timing-Dependent Plasticity
(STDP) or other local learning rules. Integrating programmable learning engines

Capstone Project 2

into the Processing Elements (PEs) would enable online adaptation and move the
architecture closer to fully neuromorphic systems.
Second, the scalability of the CGRA can be further investigated. While the
present design employs a fixed 4×4 mesh of PEs, future versions could explore
parameterizable array sizes and hierarchical interconnects to support larger neural
layers. Additionally, adaptive routing strategies or hybrid static–dynamic routing
could be evaluated to improve performance for irregular or large-scale sparse
workloads.
Third, the current accelerator uses fixed-point arithmetic optimized for
efficiency and determinism. Future work may consider mixed-precision or
configurable precision support, allowing different layers of an SNN to trade
accuracy for energy efficiency. This would be particularly beneficial for deep or
multi-layer SNN models deployed in edge or low-power environments.
Another important direction is software and compiler support. Developing a
higher-level compiler or mapping tool that automatically translates trained SNN
models (e.g., from PyTorch-based frameworks) into CGRA configuration
bitstreams and memory layouts would significantly improve usability and reduce
development effort. Such a toolchain could also perform automated scheduling and
placement optimizations tailored to the CGRA architecture.
Finally, comprehensive performance and energy evaluation on real-world
SNN benchmarks remains an important future task. Comparing the proposed CGRA
against CPUs, GPUs, and other neuromorphic accelerators using standardized
datasets and metrics would provide deeper insights into its practical advantages and
limitations. Power measurements on hardware would further validate the suitability
of the architecture for energy-constrained edge computing applications.
In summary, these development directions highlight multiple opportunities
to extend the proposed CGRA accelerator into a more scalable, programmable, and
intelligent neuromorphic computing platform.
