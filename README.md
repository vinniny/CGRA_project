## Overview

This project implements a 4×4 Coarse-Grained Reconfigurable Array (CGRA)
accelerator on the Xilinx Zynq-7000 XC7Z020 SoC (PYNQ-Z2 board). The CGRA
offloads the fully-connected (FC) stage of CNN inference from the
Cortex-A9 to a reconfigurable mesh of processing elements with multicast
routing.

The case study is MNIST handwritten digit recognition. Conv+Pool runs on
the ARM in software; FC1 (400→64) and FC2 (64→10) run on the CGRA. The
result is a measured FC-stage speedup over the ARM software baseline at
competitive accuracy.

## Architecture & References

**CGRA** decomposes as:

- **Coarse-Grained** — every compute cell operates on whole 32-bit words
  (parameter `DATA_WIDTH=32` in `cgra_top.sv`), not on single bits like an
  FPGA LUT. The granularity is one ALU operation per cycle.
- **Reconfigurable** — each PE's behaviour for a given clock cycle is
  determined by a 64-bit configuration word read out of a 16-slot context
  BRAM. The Control Unit drives `context_pc` to step through those slots.
  Reconfiguration is per-cycle, not per-bitstream.
- **Array** — compute is spatially distributed across a 4×4 mesh of PEs
  generated by nested `genvar` loops in `cgra_array.sv`.

**Literature lineage**: the spatial-mesh + ALU-PE + per-cycle reconfig
template comes from the MorphoSys architecture (Singh et al., UCI, 2000)
and the ADRES framework (Mei et al., IMEC, 2003). Closer modern
references on edge-AI CGRAs are **OpenEdgeCGRA** (EPFL, 2023, open-source)
and **CGRA4ML** (UCLA, 2024).

**What this project adds beyond the textbook template:**

| Aspect | Typical literature CGRA | This design |
|---|---|---|
| PE pipeline | 1–2 stages | 3 stages (`decode → _r → _r2`) for higher Fmax |
| MAC width | 32-bit accumulator | **40-bit saturating** — overflow-safe across long FC reductions |
| Mesh topology | Unicast nearest-neighbour | **Multicast** — single-cycle broadcast via 4-bit `route_mask` |
| Memory hierarchy | One global SPM | Per-PE 1024-word SPM **plus** 4-bank double-buffered tile memory |
| ISA | 8–12 ops | **21 ops** including hardware LIF + RELU + MAX |
| Loop hardware | Single counter | **Two-level nested L1+L2** with zero-overhead branching |

## Hardware

### Block summary

| Module (file) | One-sentence purpose |
|---|---|
| `cgra_top.sv` | Integration shell — instantiates everything below and breaks high-fanout paths via registered stalls |
| `cgra_apb_csr.sv` | APB slave; 29 control/status registers + the Result FIFO read window; drives `irq` |
| `cgra_control_unit.sv` | 3-state FSM (IDLE → RUN → FINISH) that drives `context_pc` and the L1/L2 loop counters |
| `cgra_array.sv` + `cgra_tile.sv` | 4×4 mesh wiring; each tile wraps one PE and broadcasts `alu_result` to N/E/S/W |
| `cgra_pe.sv` | The compute element: 3-stage pipeline, 21-op ISA, 40-bit MAC, 16-entry RF, 1024-word SPM, 16-slot config BRAM |
| `cgra_tile_memory.sv` | 4 row-banked BRAMs streaming activations to the west edge of each PE row; double-buffered |
| `cgra_config_broadcaster.sv` | Replays slot-0 config writes to slots 1–15 → 15× DDR bandwidth saving |
| `cgra_dma_subsystem.sv` | Wraps the engine + chain controller behind a single AXI4 master port |
| `cgra_dma_engine.sv` | Two FSMs (read + write) connected by a 64-word FIFO; decodes dst-address prefix to route to tile/config/SPM/DDR |
| `cgra_dma_chain_ctrl.sv` | 5-state FSM that walks a descriptor linked-list in DDR for scatter-gather DMA |
| `cgra_result_fifo.sv` | 4 BRAM banks × 256 entries; lockstep push from all 4 PE rows; pre-fetch register for non-destructive reads |

### Headline features

- 4×4 mesh of identical processing elements (16 PEs total)
- Each PE: 3-stage pipeline, 21-opcode ISA, 40-bit saturating accumulator,
  16-entry register file, 1024-word scratchpad, 16-slot configuration BRAM
- INT32 scalar plus INT16×2 and INT8×4 SIMD modes
- Multicast NoC: 5-bit route mask drives single-cycle broadcast to any
  combination of N/E/S/W neighbours
- Two-level hardware loop counters (L1 inner, L2 outer) with zero-overhead
  branching and a watchdog timeout
- Double-buffered tile memory (4 banks × 4096 × 32 bits) for zero-stall
  layer reconfiguration
- Scatter-gather AXI4 DMA with descriptor-chain weight movement

### Configuration paths

There are **three distinct "configuration" mechanisms** in the design,
each operating at a different level of granularity:

1. **CSR programming** — ARM writes to APB registers (`cgra_apb_csr.sv`).
   2-cycle APB transactions, zero wait-state. Sets DMA src/dst/size, loop
   bounds, auto-inc enables, IRQ mask. This is the slowest path but the
   one the ARM uses to set up each operation.
2. **PE program loading** — ARM points DMA at a 64-bit-word program in
   DDR and triggers a DMA with `DMA_DST = 0x2000_xxxx` (CONFIG prefix).
   The DMA writes word 0 to slot 0 of the target PE's config BRAM; the
   **config broadcaster** then replays the same word to slots 1–15 over
   the following 15 cycles. One DDR fetch loads all 16 slots — the
   broadcaster gives 15× DDR-bandwidth savings on single-kernel kernels.
3. **Per-cycle context selection** — Once the BRAMs are loaded, the
   Control Unit's `context_pc` indexes into each PE's config BRAM every
   cycle. Slot 0 reads one config word, slot 1 reads the next, etc. The
   PEs read whichever slot the CU points at. No DMA traffic at runtime.

### Data-flow walkthrough — one FC1 group (4 neurons in parallel)

End-to-end execution sequence for a single FC group (referenced from
`cnn_fc1_run_group` in `07_sw/baremetal/cgra_kernels_cnn.h` + the design
doc Section 11):

**Phase A — ARM configures CSRs over APB:**

```c
cgra_wr(CGRA_DMA_SRC,        DDR_WEIGHTS_ADDR);    // 0x08
cgra_wr(CGRA_DMA_DST,        0x40001234);          // 0x0C: 0x4xxx prefix → SPM
cgra_wr(CGRA_DMA_SIZE,       1600);                 // 0x10: 400 × 4 bytes
cgra_wr(CGRA_LOOP_START,     0);                    // 0x48
cgra_wr(CGRA_LOOP_END,       15);                   // 0x4C: full 16-slot context
cgra_wr(CGRA_LOOP_COUNT,     399);                  // 0x50: 400 outer iterations
cgra_wr(CGRA_SPM_AUTO_INC,   1);                    // 0x84
```

**Phase B — kick the DMA, then the CU:**

```c
cgra_wr(CGRA_DMA_CTRL, 1);     // 0x00 bit[0] = dma_start (auto-clears)
// (wait for DMA done via IRQ or polling DMA_STATUS bit[1])
cgra_wr(CGRA_CU_CTRL,  1);     // 0x20 bit[0] = cu_start (auto-clears)
```

**Phase C — what the hardware does:**

- DMA reads 400×4 = 1,600 weight bytes from DDR over AXI4 bursts (clamped
  to 4 KB page boundaries).
- The destination prefix `0x4xxx` routes the engine to per-PE SPMs.
  Offset bits `[15:12]` select PE 0–3 (this group's four PEs), bits
  `[11:0]` are the SPM word address.
- After DMA done, ARM flips `TILE_BANK_SEL` and triggers `cu_start`.
- The CU runs the 16-slot context program 400 times (`LOOP_COUNT=399`):
  slot 0 = `ACC_CLR`, slots 1–15 = `MAC` with `SPM_AUTO_INC` advancing
  the weight pointer by 16 every loop wrap.
- The Tile Memory streams activations to the west edge of row 0–3. Each
  PE reads `SPM[imm + spm_iter_cnt + context_pc]` for weights and the
  west-input register for the activation. The 40-bit MAC accumulator
  feeds back combinationally each cycle.
- After 400 iterations, the CU asserts `array_done` (a 5-condition AND);
  the FSM transitions RUN → FINISH, fires `cu_done`, and asserts `irq`.

**Phase D — ARM reads results:**

```c
// Wait for IRQ or poll CU_STATUS[1]
uint32_t r0 = cgra_rd(CGRA_RESULT_ROW0);   // 0x58: PE row 0 accumulator
uint32_t r1 = cgra_rd(CGRA_RESULT_ROW1);   // 0x5C
uint32_t r2 = cgra_rd(CGRA_RESULT_ROW2);   // 0x60
uint32_t r3 = cgra_rd(CGRA_RESULT_ROW3);   // 0x64
cgra_wr(CGRA_RESULT_POP, 1);               // 0x88: advance FIFO pointer
// then add bias, ReLU, requantize → next 4 elements of act64
```

### Interrupt & completion handshake

The CSR maintains three sticky latches and OR's their masked outputs
into the `irq` line (`cgra_apb_csr.sv:389–391`):

```systemverilog
irq <= |(reg_irq_status & reg_irq_mask);   // registered 1 cycle
```

| Source pulse | Latched into | Status bit | Mask bit |
|---|---|---|---|
| `dma_done_i` | `dma_done_latch` | `IRQ_STATUS[0]` | `IRQ_MASK[0]` |
| `cu_done_i` | `cu_done_latch` | `IRQ_STATUS[1]` | `IRQ_MASK[1]` |
| `dma_error_valid_i` (rising edge) | `dma_error_latch` | `IRQ_STATUS[2]` | `IRQ_MASK[2]` |

The `irq` net runs to the Zynq PS7 GIC. On the ARM side the bare-metal
GIC driver (`07_sw/baremetal/gic.{c,h}`) registers an ISR for the CGRA
interrupt ID and the CPU sleeps with `wfi` until the GIC wakes it.

**Clear sequence** (ISR body):

```c
uint32_t st = cgra_rd(CGRA_IRQ_STATUS);   // 0x30
if (st & 0x2) {
    /* handle CU done — read results from RESULT_ROWx */
    cgra_wr(CGRA_IRQ_STATUS, 0x2);        // W1C: write 1 to bit[1]
}
if (st & 0x1) {
    cgra_wr(CGRA_IRQ_STATUS, 0x1);        // W1C: write 1 to bit[0]
}
```

`IRQ_STATUS` is **W1C** (write-1-to-clear) — writing a 1 to any bit
clears that bit's latch; writing a 0 has no effect. Engine "set" wins
over software "clear", so if `cu_done_i` pulses on the same cycle the
ARM W1C-writes the register, the latch stays set and no interrupt is
lost (`cgra_apb_csr.sv:225–227`).

The `DMA_ERROR` register (`0x38`) is separately sticky and stays
readable even after `IRQ_STATUS[2]` is cleared, so the ARM can decode
the AXI BRESP/RRESP code; it clears only on the next `dma_start`.

### Result read window (decoded inside `cgra_apb_csr.sv`)

These addresses are part of the public APB map and the read/write
decode lives inside `cgra_apb_csr.sv` alongside every other register.
The data signals (`global_result_i`, `result_fifo_pop_data_i`, status
flags) come in as input ports from `cgra_top.sv`; the pop pulse
(`result_fifo_pop_read`) is generated from a write to `ADDR_RESULT_POP`
(`0x88`) and routed out as an output port. Reads from `0x44`
(`RESULT_STATUS`) return the FIFO status; writes to `0x44` are silent
no-ops.

| Offset | Read returns | Write effect | Notes |
|---|---|---|---|
| 0x40 | `global_result` — registered east-edge value of PE[3][3] | — | Legacy single-cell debug latch, not FIFO data |
| 0x44 | FIFO status: `[0]` pop_valid, `[9:1]` count (0–256), `[10]` overflow, `[11]` underflow | — (writes are no-ops; RO) | Pop trigger moved to 0x88 in commit refactoring the overload |
| 0x58 | `result_fifo_pop_data[0]` — pre-fetched result, PE row 0 | — | Non-destructive |
| 0x5C | `result_fifo_pop_data[1]` — pre-fetched result, PE row 1 | — | Non-destructive |
| 0x60 | `result_fifo_pop_data[2]` — pre-fetched result, PE row 2 | — | Non-destructive |
| 0x64 | `result_fifo_pop_data[3]` — pre-fetched result, PE row 3 | — | Non-destructive |
| 0x88 | reads as 0 | `RESULT_POP` — write any value to pop one entry | Newly separated from 0x44 (see Implementation Notes) |

Correct read sequence per FIFO entry: read 0x58 → 0x5C → 0x60 → 0x64
(non-destructive), then **write to 0x88** (`RESULT_POP`) to advance
the pre-fetch register. Writes to 0x44 (`RESULT_STATUS`) are no-ops.

## Implementation Notes & Known Limitations

The following items are real and documented as-is. They do not block
the demonstrated functionality; cleaning them up is left for a future
revision.

### `0x40` returns `PE[3][3]` east-edge, not "RESULT_DATA"

Address `0x40` returns `global_result`, which `cgra_top.sv:737–744`
defines as the registered east-edge output of PE row 3 (the bottom-row
rightmost cell). This is a single-cell debug latch left over from an
earlier scalar-result API and is unrelated to the four-row Result FIFO.

The address is still mapped because removing it would break any test
or demo that polls `0x40` expecting a result-shaped value. A future
rename would be either `PE3_EAST_LAST` (descriptive) or retirement
(retire the address and reclaim the offset).

### Convolution stage runs on ARM, not the CGRA

The MNIST demo's Conv1 and Conv2 layers execute on the Cortex-A9 with
VFP, not on the CGRA. The reason is documented in the `cnn_conv3x3_pixel`
kernel: the 9-MAC reduction in a 3×3 convolution saturates the INT32
accumulator on realistic input magnitudes, dropping accuracy below the
97% target. Widening the result-readout path so that Conv can keep its
40-bit partial-sum precision is listed as future work in the thesis.

The FC stage — which dominates per-image runtime — is fully on the
CGRA and delivers the 3.81× speedup that drives the headline result.

### MAC pipeline RF read-after-write hazard

The 3-stage PE pipeline (decode → `_r` → `_r2`) has no register-file
bypass. A dependent RF read in slot N+1 sees stale data because slot
N's writeback has not landed yet. Effective MAC throughput is roughly
86% (13 of 15 back-to-back MACs contribute) on the canonical FC kernel
program. Routing through neighbour mesh ports or inserting spacing
slots is the documented workaround.
